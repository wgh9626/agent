# 职责说明

## 概述

本项目将软件交付过程拆分为两个 Agent 角色：

1. Java 代码 Agent
2. Java CI/CD Agent

这样设计的目的，是把“业务代码生成”与“构建、镜像发布、部署、环境验证”解耦，避免职责混杂。

---

## 角色一：Java 代码 Agent

### 主要职责

Java 代码 Agent 负责：

- 创建和修改 Java 业务代码
- 更新应用源码仓库中的项目文件
- 维护让项目能够编译通过所需的代码结构、依赖与配置
- 产出一个可交付的源码版本

### 允许修改的范围

Java 代码 Agent 可以修改：

- `src/main/java/**`
- `src/main/resources/**`
- `src/test/**`
- `pom.xml`
- `Dockerfile`（当应用变更确实需要时）
- 其他为了保证项目可构建而必须调整的源码仓库文件

### 不负责的内容

Java 代码 Agent 不负责：

- 推送容器镜像到 Harbor
- 执行 Kubernetes 部署
- 更新 GitOps 部署仓库
- 触发或处理 Argo CD 同步
- 宣布发布成功或失败
- 未经明确要求擅自修改运行环境配置

### 交付输出

Java 代码 Agent 向下游交付时，应提供：

- 源码仓库地址
- git ref（分支、标签或提交哈希）
- 服务名称
- 构建时需要注意的补充说明（如 JDK 版本、Maven profile 等）

---

## 角色二：Java CI/CD Agent

### 主要职责

Java CI/CD Agent 负责：

- 接收一个已存在的 Java 项目发布请求
- 检出指定的仓库与代码版本
- 使用 Maven 构建 Java 制品
- 构建并打标签容器镜像
- 将镜像推送到 Harbor
- 更新部署清单或 GitOps 仓库中的镜像引用
- 触发或等待部署收敛
- 验证 rollout、Pod Ready 状态与健康检查结果
- 返回结构化的发布结果

### 允许执行的范围

Java CI/CD Agent 可以：

- 读取源码仓库
- 执行构建脚本
- 构建并推送镜像
- 更新 Kubernetes YAML 中的镜像引用
- 提交并推送 GitOps 仓库变更
- 查询 Kubernetes rollout 状态
- 查询 Argo CD 应用状态
- 在策略允许或请求明确开启时执行回滚

### 不负责的内容

Java CI/CD Agent 不负责：

- 虚构或重写业务需求
- 在发布过程中随意改业务代码
- 悄悄修改部署拓扑或集群策略
- 在未完成验证前宣称发布成功

---

## 仓库职责划分

### 应用源码仓库

示例：
- `demo-java-service`

用途：
- 存放 Java 源代码
- 存放构建文件
- 存放 Docker 构建上下文
- 若走直发模式，也可存放本地部署 YAML

主要写入者：
- Java 代码 Agent
- 人类开发者

CI/CD 使用方式：
- 检出代码
- 执行构建
- 构建镜像

---

### GitOps 仓库

示例：
- `demo-java-service-gitops`

用途：
- 存放 Kubernetes 部署清单
- 存放环境期望状态
- 存放供 Argo CD 使用的镜像版本引用

主要写入者：
- Java CI/CD Agent
- 平台/运维维护者

约束：
- Java 代码 Agent 不应直接修改该仓库中的发布镜像版本

---

## Agent 交接边界

两个 Agent 的交接点发生在：

- 业务代码已准备完成
- 已有稳定的 git ref 可以作为发布输入

### 上游应交付的最小信息

- `service_name`
- `git_repo`
- `git_ref`
- `build_path`
- `dockerfile_path`
- 可选的构建说明

### 从交接点开始，由 Java CI/CD Agent 负责

- build
- image
- deploy
- verify
- 产出 release result

---

## 部署模式

### 直发模式（direct）

Java CI/CD Agent 直接使用部署 YAML 和 `kubectl apply` 将服务发布到 Kubernetes。

### GitOps 模式（gitops）

Java CI/CD Agent 更新 GitOps 仓库中的镜像引用，再由 Argo CD 负责将变更同步到 Kubernetes。

---

## 成功标准

一次发布只有在以下条件全部满足时，才可视为成功：

1. 代码检出成功
2. Maven 构建成功
3. 容器镜像构建成功
4. Harbor 推送成功
5. 部署执行或 GitOps 收敛成功
6. Rollout 状态成功
7. Pod 进入 Ready
8. 健康检查返回成功

任意一步失败，都必须将结果标记为失败。

---

## 审计要求

每次发布都应记录：

- 服务名称
- git ref
- 镜像仓库与标签
- 部署模式
- 命名空间
- 执行到的阶段
- 最终状态
- 相关提交哈希
- 如发生回滚，记录回滚状态
