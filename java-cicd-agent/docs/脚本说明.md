# 脚本说明

## 总入口

### `scripts/release.sh`

这是 Java CI/CD Agent 的统一执行入口，用于串联：

1. Maven 构建
2. 镜像构建
3. 镜像推送
4. 部署执行
5. 发布验证
6. 失败时按策略回滚
7. 输出结构化 JSON 结果

### `scripts/run_release_from_request.sh`

这是请求文件入口脚本，用于：

1. 读取 `release-request.json`
2. 解析脚本支持的字段
3. 校验关键字段
4. 准备源码仓库
5. 在 GitOps 模式下准备 GitOps 仓库
6. 调用 `release.sh`

---

## 主要脚本

### `prepare_source.sh`
- 按仓库地址和 git ref 准备本地源码目录
- 首次执行时 clone，后续复用目录并 fetch
- checkout 指定版本；若存在同名远端分支则 reset 到 `origin/<git_ref>`

### `build_java.sh`
- 校验构建目录
- 校验 `pom.xml`
- 执行 Maven 构建

### `build_image.sh`
- 校验 Dockerfile
- 执行 `docker build`

### `push_image.sh`
- 执行 `docker push`

### `deploy_k8s.sh`
- 直发模式下执行 `kubectl apply`

### `update_gitops_image.sh`
- GitOps 模式下更新 deployment YAML 中的镜像引用
- 自动执行 `git add / commit / push`

### `check_argocd_app.sh`
- 等待 Argo CD 应用同步完成
- 检查应用健康状态

### `verify_deploy.sh`
- 检查 rollout 状态
- 检查 healthcheck 接口

### `rollback.sh`
- 当部署失败且策略允许时，执行 Deployment 回滚

## `run_release_from_request.sh` 当前支持的请求字段

会解析这些字段：

- `service_name`
- `git_repo`
- `git_ref`
- `build_path`
- `dockerfile_path`
- `image_repo`
- `image_tag`
- `deploy_mode`
- `deployment_name`
- `namespace`
- `healthcheck_url`
- `rollback_on_fail`
- `manifest_path`
- `gitops_repo`
- `gitops_branch`
- `gitops_manifest_path`
- `container_name`
- `argocd_app_name`
- `build_notes`（会读取，但不参与实际执行参数）

说明：
- 当前**不支持**从请求文件读取 `timeout_seconds`
- 入口脚本会固定把 `300` 作为 `release.sh` 的超时时间参数

## `release.sh` 参数说明

用法：

```sh
release.sh \
  <service_name> \
  <git_ref> \
  <repo_dir> \
  <build_path> \
  <dockerfile_path> \
  <image_repo> \
  <image_tag> \
  <deploy_mode> \
  <deployment_name> \
  <namespace> \
  <healthcheck_url> \
  <timeout_seconds> \
  [manifest_path] \
  [gitops_repo_dir] \
  [gitops_manifest_path] \
  [container_name] \
  [argocd_app_name] \
  [rollback_on_fail]
```

### 关键参数

- `service_name`：服务名
- `git_ref`：本次发布的源码版本标识
- `repo_dir`：源码仓库本地路径
- `build_path`：Maven 构建目录
- `dockerfile_path`：Dockerfile 路径
- `image_repo`：镜像仓库
- `image_tag`：镜像标签
- `deploy_mode`：`direct` 或 `gitops`
- `deployment_name`：Kubernetes Deployment 名称
- `namespace`：目标命名空间，默认建议为 `test`
- `healthcheck_url`：健康检查地址
- `timeout_seconds`：验证超时时间

### direct 模式额外参数

- `manifest_path`：源码仓库中的部署清单路径

### gitops 模式额外参数

- `gitops_repo_dir`
- `gitops_manifest_path`
- `container_name`
- `argocd_app_name`（可选；提供后会触发 Argo CD 检查）

---

## `release.sh` 输出字段

当前脚本在成功或失败时输出的核心字段为：

- `service_name`
- `git_ref`
- `image`
- `deploy_mode`
- `namespace`
- `stage_reached`
- `status`
- `build_result`
- `image_push_result`
- `deploy_result`
- `rollout_result`
- `health_check_result`
- `rollback_performed`
- `next_action`

在特定情况下还会输出：

- `argocd_app`
- `error_summary`
- `error_excerpt`

说明：
- 当前结果中**没有** `success`、`stage`、`gitops_commit`、`git_repo` 等字段
- 结果消费方应以当前真实输出为准，而不是按理想字段脑补

## 当前实现说明

当前脚本版本适合：

- 单服务
- `test` 环境优先
- 结构较稳定的 Deployment YAML
- 先跑通最小闭环

后续建议增强：

- 更稳健的 YAML 修改方式（如 `yq` 或 `ruamel.yaml`）
- 更详细的阶段日志持久化
- 对接 Jenkins 参数化构建
- 对接 Argo CD revision 校验
- 支持多环境与更细粒度回滚策略

---

## 一句话理解

`run_release_from_request.sh` 负责把请求文件翻译成命令行参数，`release.sh` 负责真正执行发布并吐出结构化结果；两者合起来才是这套发布链的真实入口。
