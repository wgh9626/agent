# 请求文件说明

## 目的

`release-request.json` 用于让上游 Agent、Jenkins、人工脚本或其他自动化入口，以统一格式调用 `scripts/run_release_from_request.sh`。

它的作用是把“要发布什么、如何发布、发布到哪里”描述清楚，避免靠临时口头约定传参。

---

## 推荐调用链路

```text
release-request.json
  ↓
run_release_from_request.sh
  ↓
prepare_source.sh
  ↓
release.sh
  ↓
结构化 JSON 结果（标准输出）
```

---

## 字段说明

### 基础字段（当前脚本必填）

- `service_name`：服务名称
- `git_repo`：源码仓库地址
- `git_ref`：本次要发布的分支、标签或提交哈希
- `build_path`：Maven 构建路径
- `dockerfile_path`：Dockerfile 路径
- `image_repo`：镜像仓库地址
- `image_tag`：镜像标签
- `deploy_mode`：部署模式，可选 `direct` 或 `gitops`
- `deployment_name`：Kubernetes Deployment 名称
- `namespace`：目标命名空间，默认建议为 `test`
- `healthcheck_url`：健康检查地址

### 通用可选字段

- `rollback_on_fail`：失败时是否允许回滚
- `build_notes`：仅供人阅读的补充说明，不参与执行逻辑

### direct 模式字段

当 `deploy_mode=direct` 时，必须提供：

- `manifest_path`：源码仓库中的部署清单路径

### gitops 模式字段

当 `deploy_mode=gitops` 时，必须提供：

- `gitops_repo`：GitOps 仓库地址
- `gitops_manifest_path`：GitOps 仓库中的 Deployment YAML 路径
- `container_name`：Deployment 中要更新镜像的容器名称

可选提供：

- `gitops_branch`：GitOps 仓库分支，不填时默认 `main`
- `argocd_app_name`：对应的 Argo CD Application 名称；提供后会触发额外状态检查

```json
{

  "git_repo": "git@example.com:team/demo-java-service.git",

  "dockerfile_path": "./Dockerfile",
  "image_repo": "harbor.example.com/test/demo-java-service",
  "image_tag": "20260326-1045-abcd123",
  "deploy_mode": "gitops",
  "manifest_path": "",
  "gitops_repo": "git@example.com:team/demo-java-service-gitops.git",
  "gitops_branch": "main",
  "gitops_manifest_path": "apps/test/deployment.yaml",
  "deployment_name": "demo-java-service",

  "namespace": "test",
  "healthcheck_url": "http://demo-java-service.test.svc.cluster.local/actuator/health",

  "rollback_on_fail": true,
  "build_notes": [
    "需要 JDK 17",
    "使用 Maven profile: test"
  ]
}
```

## 调用方式

```sh
sh scripts/run_release_from_request.sh templates/release-request.json
```

也可以指定源码缓存目录：

```sh
sh scripts/run_release_from_request.sh templates/release-request.json /tmp/java-cicd-agent-sources
```

---

## 当前实现说明

当前版本会：

1. 读取 JSON 请求文件
2. 校验关键字段
3. 自动准备源码仓库
4. 若是 GitOps 模式，再准备 GitOps 仓库
5. 调用 `release.sh` 执行完整发布流程

注意：
- 当前版本会固定把 `300` 作为超时时间传给 `release.sh`
- 当前请求文件**不支持** `timeout_seconds` 字段
- 当前版本适合单服务、单请求、单次执行
- 更复杂的批量发布、矩阵环境、多服务编排，可以后续再扩展

## 一句话理解

这个请求文件不是“随便凑个 JSON”，而是 `run_release_from_request.sh` 的输入契约；写什么能生效，要以脚本真实解析逻辑为准。
