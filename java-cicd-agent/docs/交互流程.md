# 交互流程

## 总体流程

1. Java 代码 Agent 完成业务代码修改。
2. Java 代码 Agent 提交代码，或明确给出可发布的 git ref。
3. Java 代码 Agent 向 Java CI/CD Agent 提交发布请求。
4. Java CI/CD Agent 校验请求参数是否完整。
5. Java CI/CD Agent 检出源码并执行 Maven 构建。
6. Java CI/CD Agent 构建镜像并推送至 Harbor。
7. Java CI/CD Agent 根据发布模式执行后续动作：
   - `direct`：直接应用 Kubernetes YAML
   - `gitops`：更新 GitOps 仓库镜像标签并推送变更
8. Java CI/CD Agent 等待部署完成。
9. Java CI/CD Agent 检查滚动发布状态、Pod Ready、健康检查。
10. Java CI/CD Agent 返回结构化发布结果。

## 失败路由

### 1. 代码构建失败

说明：
- 依赖缺失
- 编译报错
- 单元测试失败
- Maven profile 不正确

处理建议：
- 返回失败结果给上游 Java 代码 Agent 或开发者
- 明确指出失败阶段为 `build`
- 附带关键错误片段

### 2. 镜像构建失败

说明：
- Dockerfile 路径错误
- 构建上下文缺失
- 基础镜像拉取失败

处理建议：
- 返回失败结果
- 明确指出失败阶段为 `image`
- 给出 Docker 构建错误摘要

### 3. 镜像推送失败

说明：
- Harbor 凭据错误
- 仓库不存在
- 网络失败

处理建议：
- 返回失败结果
- 明确指出失败阶段为 `push`
- 给出凭据、仓库或网络相关提示

### 4. 部署失败

说明：
- YAML 不合法
- Deployment 名称错误
- 命名空间错误
- Argo CD 未同步

处理建议：
- 返回失败结果
- 明确指出失败阶段为 `deploy`
- 若允许回滚，则按策略决定是否回滚

### 5. 健康检查失败

说明：
- rollout 成功但服务未真正可用
- 应用启动异常
- 依赖服务未就绪

处理建议：
- 结果必须判定为失败
- 明确指出失败阶段为 `verify`
- 不得因 `kubectl apply` 成功就宣称发布成功

---

## 协作原则

1. Java 代码 Agent 只负责业务代码与源码仓库内容。
2. Java CI/CD Agent 只负责交付、部署与验证。
3. GitOps 仓库中的镜像版本应由 Java CI/CD Agent 统一更新。
4. 发布成功必须以完整验证通过为准，而不是以“命令执行成功”为准。
5. 所有发布结果都应结构化输出，便于审计与自动化处理。
